# Развёртывание и безопасность

Приложение рассчитано на работу в интернете **только** при корректных переменных окружения. Без них контейнер/процесс при старте завершится с ошибкой (намеренно).

## Обязательные настройки (продакшен)

| Переменная | Назначение |
|------------|------------|
| `SECRET_KEY` | Секрет подписи сессии и CSRF. **Один и тот же** для всех воркеров Gunicorn/uwsgi. |
| `SITE_READ_PASSWORD` | Пароль «Доступ судьи»: после входа открываются **HTML со всеми данными** (турниры, карточки, аналитика) и `/api/*`. |
| **или** `API_KEYS` | Через запятую: ключи для интеграций (`Authorization: Bearer <ключ>` или `X-API-Key`). Без `SITE_READ_PASSWORD` HTML-гейт выключен, но закрыт только `/api/*`. |

Аварийное открытие HTML для всех без входа (не для интернета): **`DISABLE_PUBLIC_HTML_GATE=1`** — сохраняет пароль в конфиге, но страницы снова видны гостям.

Дополнительно для HTTPS за прокси:

| Переменная | Назначение |
|------------|------------|
| `SESSION_COOKIE_SECURE=1` | Cookie сессии только по HTTPS. |

База данных: задаётся **`DATABASE_URL`** или **`SQLALCHEMY_DATABASE_URI`** (если заданы обе, используется `DATABASE_URL`).

## Только для разработки (не в интернете)

- `ALLOW_INSECURE_DEFAULTS=1` — ослабляет проверки при старте, генерирует временный `SECRET_KEY`, если он не задан.
- `DISABLE_PUBLIC_API_AUTH=1` — **полностью открывает** все `/api/*` без ключа и без сессии (кроме мониторинга не использовать в продакшене).
- `DISABLE_PUBLIC_HTML_GATE=1` — отменяет обязательный вход судьи/админа для HTML (данные снова видны всем).
- `FLASK_DEBUG=1` — то же предупреждение, что и для ослабленного режима.

## CORS

По умолчанию cross-origin CORS **выключен**. Чтобы разрешить конкретным сайтам доступ к API с учётом cookie (редко нужно), задайте:

`CORS_ORIGINS=https://example.com,https://app.example.com`

## Эндпойнты

- `GET /api/health` — без авторизации (проверка «жив ли процесс»).

Остальные `GET/POST /api/...` требуют одного из:

- сессии администратора;
- сессии судьи после `/site-access` с верным `SITE_READ_PASSWORD`;
- валидного ключа из `API_KEYS`.

## Поведение при обновлении

После включения защиты анонимные запросы к JSON и просмотр страниц с данными без входа недоступны. Судьи один раз проходят **«Доступ судьи»**; администратор — через обычный вход.

При входе администратора сессия очищается (защита от фиксации сессии) — незавершённый мастер загрузки XML нужно пройти заново.

## Резервные копии

Политики у вас может не быть — минимальный практический минимум: ежедневный снимок файла БД и каталога `uploads` вне сервера приложения; проверка восстановления раз в квартал.
